## 楔子：项目叫 AI Agent，但其实只是 RAG 检索器

v2.1 完工那天我写了句反思：

> 我这个项目的名字叫 **AI Collector**，定位写的是 **"AI Agent"**。
> 但实际架构是：监控 → 采集 → LLM 清洗 → 入库 → 向量检索。
> LLM 只是"被调用的函数"，没有决策权。

到了 v2.1 我加了 RAG，能用自然语言"问"数据了。但**依然没有 Agent 的灵魂——它不会自己想办法，只会忠实执行**。问 "薪资 20K+ 的 AI 岗"，它就老老实实搜，搜不到就给空结果，**不会主动换关键词、不会反思、不会替你想替代方案**。

这一篇要做的事是：

**把它从"RAG 检索器"真正升级成"AI Agent"**——能听人话、能自主决策、能反思、能给替代方案。

而且整件事有一个**双倍闭环的好玩之处**：

- 用 v3.0 这个 Agent，**给我自己找 AI 工作**
- 简历上能写"用自己造的 Agent 找到这份工作"——比任何技术堆砌都更打动面试官

这不是炫技，是现实——我刚被医生确诊糖尿病前期，准备辞掉做了两年的第三方测评工作，转 AI Agent 开发。9-10 月要在新城市拿到 offer。**我需要一个能每天扫五个城市真实 AI 岗位、按我画像打分、告诉我该学什么的工具**。

那就自己造一个。

时间：一天（实际编码 6 小时）。

## 一、v3.0 的"三件套"和那道分水岭

打开 v2.1 的代码，画出来当前架构：

```
监控（Bilibili / arXiv 源） → 采集 → LLM 清洗 → SQLite
                                                  ↓
                                            bge-m3 embed
                                                  ↓
                                            Milvus 向量库
                                                  ↓
                                         自然语言查询（RAG）
```

这是一个 **AI Pipeline**。每个环节都是确定性的：「监控完一定采集，采集完一定清洗，清洗完一定入库」。LLM 只在"清洗"那一步被调用一次，**没有任何决策权**。

要升级成 Agent，需要补这三件：

### 能力 1 ── 自主采集（Agent 自己找 URL，不是人喂）

- 新增 `BossSource`：Boss 直聘招聘源
- 与 v2.0 的 `BilibiliSource`、`ArxivSource` 同架构（一个抽象接对所有源）
- 复用所有反爬基建（Playwright + stealth）

### 能力 2 ── 自主决策（LangGraph 编排）

- 节点 1：理解用户需求（"15K+"、"5 城市"、"避开北京" 等约束）
- 节点 2：决策搜什么、怎么搜
- 节点 3：调用工具采集
- 节点 4：反思是否充分（信息不够 → 换关键词重搜）
- 节点 5：综合输出 + 引用每条数据源

### 能力 3 ── 自主匹配（Profile-aware）

- `my_profile.yaml`：学历、年限、技能、目标
- `match_score(jd, profile)`：技能 overlap + gap 分析
- 输出："你已具备 X，需补 Y，建议优先级 Z"

**核心区别**——

```
Pipeline:     固定步骤，LLM 是工具
Agent:        LLM 是主控大脑，自己决定下一步做什么、用什么工具
```

简历上写「AI Agent 项目」，面试官真问一句"Agent 体现在哪？"，**v2.1 我只能答"用了 LLM 做 JSON 抽取"——撑不起 Agent 这个词**。v3.0 完成后，我能答："基于 LangGraph 编排的多节点决策流，含反思循环，能根据失败结果自主换关键词重搜，最终给出错配诊断和替代策略。"

这就是分水岭。

## 二、Phase 1：摸清 Boss 直聘的反爬画像

按 v2.0 博客确立的"vertical tracer bullet"原则，先**不写代码，先摸现状**。

打开 Boss 直聘 PC 站 `zhipin.com`，F12 看列表：

```html
<div class="job-list">
  <canvas width="800" height="600"></canvas>   <!-- ← 全是这个 -->
</div>
```

整个搜索结果列表**用 Canvas 渲染**，DOM 里没有岗位文字。常规爬虫直接废。

怎么办？三条路径：

```
路径 A: 接口逆向（最优雅）
  Canvas 数据来自某个 XHR/WS → 抓 JSON
  难度：🟡 中

路径 B: Canvas 截图 + OCR（暴力但稳）
  Playwright 截图 → 调 OCR
  难度：🟢 低
  缺点：慢、贵

路径 C: 移动端 H5（取巧）
  Boss 移动端 m.zhipin.com 不是 Canvas，是 DOM
  难度：🟢 低
```

我去试 C —— **移动端 H5 浏览器开发者模式直接看 Network**，立刻发现一个 JSON 接口：

```
GET m.zhipin.com/wapi/zpgeek/search/joblist.json
    ?query=AI应用开发&city=101210100&page=1
```

返回结构化岗位列表（jobName / salaryDesc / skills / jobLabels / 发布人）。**比 Canvas、比 DOM 解析都简单。**

但是——

```bash
$ curl 'https://m.zhipin.com/wapi/zpgeek/search/joblist.json?...'
{"code": 37, "message": "您的环境存在异常.", "zpData": {"seed": "...", "name": "..."}}
```

风控 code 37。`seed` + `name` 这套是 Boss 的"环境签名"。**社区里有逆向方案**，但要逆向 `__zp_stoken__` 算法——半天就没了。

**关键洞察**：

> 我不需要逆向。我已经有一个登录态的 Chrome 在跑了——直接让 **Playwright 接管真实浏览器**，在已登录的 m.zhipin.com 页面里同源 fetch 这个 API。

cookie 浏览器自动带，签名是浏览器自动算的，**风控直接绕开**。

方案敲定：**CDP（Chrome DevTools Protocol）接管 Chrome**。

### 必踩的两个坑

**坑 1**：Chrome 安全策略不允许默认 profile 开调试端口。

试了一下退出 Chrome → 用 `--remote-debugging-port=9222` 重启，结果端口没监听。看 Chrome 日志：

```
DevTools remote debugging requires a non-default data directory.
Specify this using --user-data-dir.
```

原因：现在 Chrome 出于安全考虑，不允许在你日常 profile 上开调试端口——防止恶意脚本读你日常浏览数据。

**解法**：用独立调试 profile。

```bash
mkdir -p ~/.hermes/chrome-debug-profile

osascript -e 'quit app "Google Chrome"' && sleep 2 && \
/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9222 \
  --user-data-dir="$HOME/.hermes/chrome-debug-profile" \
  https://m.zhipin.com \
  > /tmp/chrome-debug.log 2>&1 &
```

这样**日常 Chrome 和调试 Chrome 完全隔离**，调试 profile 里扫一次码持久保存，下次启动不用重扫。

**坑 2**：用错 Python。

`playwright.async_api` 装在 pyenv 3.11.9 上，系统 `/usr/bin/python3` 里没有。第一次跑：

```
ModuleNotFoundError: No module named 'playwright'
```

解：脚本前面 `eval "$(pyenv init -)"`，确保用 pyenv 那个 python。

## 三、Phase 2：BossSource 插件式接入

抽象出 `BaseSource` 已经是 v2.0 做过的事。Phase 2 的真正考验是：**Boss 这个全新源接进来时，v2.0 的抽象站不站得住？**

答案是：**站得住，只改 main.py 一行**。

### BossSource 的核心代码

```python
class BossSource(BaseSource):
    source_type: str = "boss_zhipin"

    async def fetch_jobs_structured(self) -> List[BossJob]:
        async with async_playwright() as p:
            browser = await p.chromium.connect_over_cdp(self.cdp_url)
            target_page = await self._get_or_create_zhipin_page(browser)

            for city in self.cities:
                for keyword in self.keywords:
                    jobs = await self._fetch_with_retry(
                        target_page, city=city, keyword=keyword, page_num=1
                    )
                    all_jobs.extend(jobs)
                    await asyncio.sleep(random.uniform(0.6, 1.4))  # 节流

        # 启发式去噪
        kept = [j for j in all_jobs if not j.is_likely_noise]
        return kept
```

### Bug 修复：风控是工程问题，不是逆向问题

诊断（用 systematic-debugging skill 的方法）：

**假设 A**：Boss 对"低线城市 + 高频请求"风控更严？
**假设 B**：连续高频请求被临时风控？
**假设 C**：cookie 出问题了？

最便宜的验证：**手动在浏览器地址栏直接请求郑州 URL**——立刻返回 code 0。

所以不是 cookie 问题，**是连续请求被临时风控了**。解法：

1. **每次请求间随机延迟 0.6-1.4 秒**（节流）
2. **失败请求带指数退避重试 1.5^attempt 秒，最多 2 次**

代码：

```python
async def _fetch_with_retry(self, page, *, city, keyword, page_num):
    last_status = "unknown"
    for attempt in range(self.max_retries + 1):
        jobs, status = await self._fetch_one_query(...)
        if status == "ok":
            return jobs
        last_status = status
        if attempt < self.max_retries:
            backoff = (1.5 ** attempt) + random.uniform(0.3, 0.9)
            logger.info(f"{status} → {backoff:.1f}s 后重试")
            await asyncio.sleep(backoff)
    return []
```

加完跑第二次：

```
🎉 第 2 次跑（加 retry + 节流）：
   总抓 270 条 → 去噪 238 条
   郑州 51 / 苏州 50 / 青岛 47 / 杭州 45 / 济南 45  ← 全 5 城市覆盖

   所有 code 37 → 重试 1-2 次后全部恢复
```

> 💡 **教训 2：Boss 风控是临时性的，不是持续性的。**
> 同一查询失败一次，1-2 秒后重试 80%+ 会成功。不需要换 IP、不需要逆向签名、不需要刷新 cookie。**这是个工程问题，不是逆向问题。**

### 意外发现：郑州 AI 岗位 ≥ 杭州（51 vs 45）

这一下推翻了我之前"郑州 AI 岗位稀缺"的预判——**真实数据永远比预设更值钱**。但要下定论还得看薪资分布 + 公司质量，这是 P4' 要解决的。

## 四、Phase 3：接入 v2.1 RAG —— 架构复利的证明

Phase 3 是整个 v3.0 里**写得最快的一段**，30 分钟搞定。原因很简单：

**v2.0 抽象出 BaseSource、v2.1 抽象出 VectorStore——加 Boss 这一全新源时，RAG 层零修改可复用。**

这就是 v2.0 博客金句"架构复利"的现实印证。

### 数据流的特别之处

Boss 数据流和 v2.0 的 B 站/arXiv 不一样：

```
B 站 / arXiv：监控发现 URL → 采集 HTML → LLM 清洗成 JSON → 入库
Boss：         搜索 API 本身就返回结构化数据 ──→ 直接入库
```

**搜索 API 已经给了结构化数据，跳过 collector + processor 两步**。把 BossJob 按 v2.1 `structured_json` 契约 shim 一下：

```python
def boss_job_to_structured(job: BossJob) -> dict:
    return {
        "title": job.job_name,
        "summary": f"{job.city}地区，职位「{job.job_name}」，薪资 {job.salary_desc}。",
        "key_points": list(job.job_labels),
        "tags": job.skills + [job.keyword, job.city],
        "_boss": {
            "salary_desc": job.salary_desc,
            "city": job.city,
            "encrypt_job_id": job.encrypt_job_id,
            ...
        },
    }
```

落库后**索引脚本 `scripts/index_final_results.py` 零修改**，直接跑：

## 六、Phase 5：LangGraph Agent —— v3.0 的灵魂

这是整个 v3.0 真正"配得上 AI Agent"的一步。

### 选 LangGraph 而不是 ReAct / AutoGen 的理由

```
ReAct：    单一 prompt 跑，LLM 推理 + 动作 + 观察循环
           缺点：决策不可视、调试黑盒、容易陷入死循环
           
AutoGen：  多 Agent 对话框架，适合 group chat 场景
           缺点：对求职这个单 Agent 单任务过重

LangGraph：显式有向图，每个节点一个职责
           优点：可视化、可调试、可控
           优点：求职这个场景需要"反思 → 重搜"的循环
                 LangGraph 的 conditional edge 天然支持
           优点：2026 招聘 JD 出现频率高 ⭐
```

简历加分项里，LangGraph 在 2026 写出来比 ReAct/AutoGen 更值钱。

### 5 节点 DAG

```
START
   ↓
parse_intent  ─── LLM ──→ 解析自然语言为结构化意图
   ↓
retrieve      ─── 纯函数 ──→ RAG 检索 top 30
   ↓
filter        ─── 纯函数 ──→ 按薪资/城市/学历/经验/黑名单硬过滤
   ↓
reflect       ─── LLM ──→ 决策：done 还是 retry？
   │
   │ retry（换关键词）
   └──────────────────────→ 回到 retrieve
   │
   │ done（够 ≥ 5 条 或 重试 ≥ 3 轮）
   ↓
summarize     ─── LLM ──→ 生成 markdown 推荐报告
   ↓
END
```

**关键设计：决策类调 LLM，确定性逻辑零延迟。**

5 个节点里只有 3 个调 LLM（parse_intent / reflect / summarize），retrieve + filter 走纯函数。整个 Agent 跑一次 LLM 调用 5 次。**同样的功能用单一 ReAct prompt 跑下来可能要 20+ LLM 调用 + 大量幻觉。**

### LangGraph 1.x StateGraph 关键代码

```python
g = StateGraph(AgentState)                  # TypedDict total=False
g.add_node("parse_intent", parse_intent)    # 节点：state -> partial state
g.add_node("retrieve", retrieve)
g.add_node("filter", filter_node)
g.add_node("reflect", reflect)
g.add_node("summarize", summarize)

g.add_edge(START, "parse_intent")
g.add_edge("parse_intent", "retrieve")
g.add_edge("retrieve", "filter")
g.add_edge("filter", "reflect")
g.add_conditional_edges(
    "reflect",
    route_after_reflect,                    # 路由函数：state -> 下一节点名
    {"retrieve": "retrieve", "summarize": "summarize"},
)
g.add_edge("summarize", END)

compiled = g.compile()
final_state = compiled.invoke({"query": "...", "profile": {...}})
```

## 七、复盘：今天学到的 6 件事

### 1. 选型不是技术问题，是 ROI 问题（再次印证）

- 选 LangGraph 不是因为它最强大，是因为 **2026 招聘 JD 里 LangGraph 出现频率高**
- 选 CDP 接管 Chrome 不是因为它最优雅，是因为 **5 分钟解决，省下半天逆向签名**
- 选移动端 H5 不是因为它最完整，是因为 **绕开 Canvas 反爬**

每个选型决策背后都是 **「最快到达可演示状态」+「面试加分点」** 这两条标准。

### 2. Vertical tracer bullet 在 v3.0 再次验证

5 个阶段每个都是独立 commit、独立可演示。**任何一段单独跑通就是一个完整的小成果**。

- P1 跑通 → 已经知道路径可行
- P2 跑通 → 已经有 192 条真实数据
- P3 跑通 → 已经能自然语言查询
- P4' 跑通 → 数据质量翻倍
- P5 跑通 → 真正的 Agent

**如果一锅烩**（一次写完 5 个阶段才跑），那么 Bug 3（ingest 漏存 ID）要等到 P5 跑完才暴露——那时候要回头改 P2 的代码 + 重新清库 + 重新富化。**总时间至少多一倍。**

### 3. 真实数据永远比预设更值钱

- 我以为郑州 AI 岗位稀缺 → 真实数据 51 条 ≥ 杭州 45 条
- 我以为 LangChain 是冷门技能 → 196 条岗位里 45 条直接要 LangChain
- 我以为 25K+ 在转型期不现实 → JD 正文里写「20K」「20-40K」的岗位真的有

**没跑数据之前的所有判断都是猜测**。这就是为什么 P3 之后我立刻做了 4 个真实查询验证——**让数据说话**。

### 4. 节点拆 LLM / 非 LLM 极有效

```
5 节点里：
  3 个调 LLM（parse_intent / reflect / summarize）
  2 个走纯函数（retrieve / filter）

整个 Agent 跑一次 LLM 调用 5 次
同样的功能用 ReAct 单一 prompt 跑：估计 20+ 次 LLM 调用 + 大量幻觉
```

每个能用纯函数完成的环节就用纯函数，**只在"需要决策"的地方调 LLM**。这是 LangGraph 比 ReAct 优雅的核心。

### 5. 反思节点是真 Agent 的"灵魂"

没有 reflect 节点的 Agent 不是 Agent，是带 LLM 的 Pipeline。**真正区分两者的是「失败处理能力」**：

```
Pipeline 失败：抛异常 / 返回空
Agent 失败：自我反思 + 自主调整 + 给出替代方案
```

这也是为什么我特意造了 Run 2 那个**故意找不到结果的查询**——只有验证了"找不到怎么办"，才能说"这个 Agent 真的能用"。

### 6. v3.0 终于配得上 "AI Agent" 项目名

```
v1.x：采集 + LLM 清洗                = AI Pipeline
v2.x：插件式多源 + RAG                = 可扩展的 AI Pipeline
v3.0：意图理解 + 自主决策 + 反思循环  = 真 AI Agent
       + 报告生成
```

到 v3.0 这一步，简历能这样写：

> 自研基于 LangGraph + Milvus + Playwright + bge-m3 的**求职 Agent**，
> 通过 CDP 接管真实浏览器绕过 Boss 直聘 Canvas 反爬，
> 从 5 城市采集 192 条 AI 岗位真实 JD，
> **端到端实现意图解析 → RAG 检索 → 反思决策 → 报告生成**。
> 用自己造的 Agent 找到了现在这份工作。

最后一句是简历金句的来源——**双倍闭环**。
