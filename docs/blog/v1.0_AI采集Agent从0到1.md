## 楔子：为什么不直接用 RSS？

我最早想做的事情很简单——**让 AI 帮我盯着 B 站几个 UP 主，新视频一发就给我做个摘要**。

听起来像一个 RSS 订阅器加一点 AI。但当我真的去找 RSS 服务时，发现：

- B 站官方没 RSS
- RSSHub 公开镜像几乎全部超时或 503
- 自己搭 RSSHub 需要 Redis、Node.js、Docker 一堆依赖
- 即使搞定了，RSS 也只能给我标题和链接，**视频里到底讲了什么我还是要点开看**

那一刻我意识到，我要的不是 RSS，是一个**有理解能力的助手**：

```
传统 RSS：       告诉我"有新视频"
我想要的：       告诉我"有新视频，标题是 X，UP 主是 Y，
                可能在讲 Z 这件事，关键点有 a/b/c"
```

这不是订阅器，这是 **Agent**。

于是这个项目开始了。

## 一、想清楚要做什么

在写第一行代码之前，我花了一晚上把项目分解成 4 层：

```
┌─────────────────────────────────────────┐
│              AI Pipeline                │
└─────────────────────────────────────────┘
        │
        ▼
┌──────────┐  ┌──────────┐  ┌──────────┐
│ ① Monitor │→│② Collector│→│③ Processor│
│  发现新URL │  │ 抓页面内容 │  │ LLM 提取  │
└──────────┘  └──────────┘  └──────────┘
        │           │            │
        ▼           ▼            ▼
┌─────────────────────────────────────────┐
│         ④ DBManager (SQLite)            │
│  urls_history | task_queue | ...        │
└─────────────────────────────────────────┘
```

**为什么这样分层？**

因为每一层失败的概率不同、修复的成本不同：

- Monitor 失败：B 站接口挂了 / 反爬
- Collector 失败：页面结构变了
- Processor 失败：LLM 抽风返回脏 JSON
- DB 永远不该失败

**4 层之间用 SQLite 状态机连接**：

```
PENDING → PROCESSING → COLLECTED → COMPLETED
                          ↓
                       FAILED
```

任何一步挂了，状态都留在数据库里，下次启动可以从断点续跑。这是这个架构最值钱的设计——**不是因为我多聪明，是因为我做过 2 年测试，知道"任何代码都会挂"**。

## 二、技术栈选择

这部分没什么好说的，全是踩过坑选出来的：

| 模块 | 技术 | 为什么 |
|---|---|---|
| 浏览器自动化 | Playwright + stealth | Selenium 太老、puppeteer 是 JS |
| HTML 转 Markdown | markdownify | 比手写 BeautifulSoup 提取干净 100 倍 |
| LLM | 火山方舟 Coding Plan (kimi-k2.6) | 包月套餐，比 OpenAI 便宜 90% |
| 数据库 | SQLite | 单机够用，零配置 |
| 任务调度 | Python 脚本 + 状态机 | 不用 Celery，杀鸡不用牛刀 |

特别说一下**火山方舟 Coding Plan**：

它有个**专属 base_url** `/api/coding/v3`，必须用这个 URL 才能走套餐价（约 100 元/月，无限调用）；如果用错成默认 `/api/v3`，就会按 Token 量计费——同样的代码一晚上能烧 200 块。

我第一次写错了，看到火山控制台开始扣费才发现。这是这个项目踩的第一个坑，也是我整个 v1.0 学到的最贵一课：**包月模型 ≠ 任意调用，base_url 才是计费分水岭**。

## 三、第一个真正能跑通的版本

代码大概长这样（简化）：

```python
class AIPipeline:
    def __init__(self):
        self.db = DBManager()
        self.monitor = BiliMonitor()
        self.collector = BiliCollector()
        self.processor = LLMProcessor()

    async def run(self):
        # 阶段 1：发现新 URL
        new_urls = self.monitor.sync_targets(TARGET_UIDS)

        # 阶段 2：Playwright 抓内容
        await self.stage2_collect(max_count=3)

        # 阶段 3：LLM 提取结构化数据
        self.stage3_process(max_count=3)
```

## 四、那些写进 RETROSPECTIVE 的"血泪坑"

第一天能跑出 Hello World 之后，是真正考验耐心的时候。我把今天踩过的 8 个坑按"惊吓程度"排了序：

### 坑 1：playwright-stealth 的 API 偷偷换了

教程里写的：

```python
from playwright_stealth import stealth_async
await stealth_async(page)
```

我装的最新版（2.0+）里：

```python
from playwright_stealth import Stealth
await Stealth().apply_stealth_async(page)
```

不一样。这个坑跑了半小时才发现。

**教训**：装库时永远 `pip show <package>` 看版本，再去 PyPI 看 changelog。教程都是过去时。

### 坑 2：Playwright Chromium 装不下来

```bash
playwright install chromium
# Failed to download Chromium: Connection timeout
```

国外 CDN 在国内打不开。**修法**用 npmmirror：

```bash
PLAYWRIGHT_DOWNLOAD_HOST=https://npmmirror.com/mirrors/playwright \
playwright install chromium
```

5 秒搞定。这是国内做 Playwright 项目必备配置。

### 坑 3：B 站 API 412 反爬

第一次跑 monitor，理直气壮收到：

```
412 Precondition Failed
```

B 站 2024 年起对接口加了 WBI 签名机制，没带正确签名直接拒接。

我那时候还不知道签名算法是什么，先暂时绕开（v1.0 接受 monitor 失败，只手动喂 URL），把后面的 collector 和 processor 跑通。

**这是关键决策**：不是所有坑都要立刻填。**跳过它，先看终点**。

### 坑 4：LLM 输出截断

```python
max_tokens=2000
```

看起来很合理对吧？但当视频描述很长时，LLM 写到一半就被切断，输出半个 JSON：

```json
{
  "title": "...",
  "summary": "这段总结写到一半就被截断了，最后一个字是
```

`json.loads` 直接抛 JSONDecodeError，整条任务挂掉。

我当时改成 4000，临时缓解。但**根本问题在于**：

1. 没有 JSON 修复机制
2. 没有把失败的原始 LLM 输出保存下来分析
3. 没有日志告诉我哪条失败了

这 3 个问题我留到 v1.1 再修——v1.0 阶段先专注让它"能跑"。

### 坑 5：B 站 DOM 选择器过期

我一开始按 2023 年的教程写：

```python
desc = soup.select_one(".video-desc")
```

跑出来全是 `None`。F12 看现在的 DOM——B 站早就改成了：

```python
desc = soup.select_one(".video-desc-container")  # 或 #v_desc
```

**永远不要相信教程里的 CSS 选择器**。前端改版从来不通知。

### 坑 6-8：GitHub Token 权限 / RSSHub Redis bug / 计费陷阱

篇幅原因省略，详见 GitHub 仓库的 [RETROSPECTIVE.md](https://github.com/nakajimamiyuki/ai_collector_project/blob/main/RETROSPECTIVE.md)。

## 五、跑通后的成就感

当我看到第一条数据从数据库里读出来：

```json
{
  "title": "【官方 MV】Never Gonna Give You Up - Rick Astley",
  "tags": ["欧美音乐", "MV", "Rickroll", ...],
  "summary": "英国歌手 Rick Astley 经典代表作《Never Gonna Give You Up》的官方 MV...",
  "key_points": [
    "Rick Astley 经典代表作官方 MV",
    "欧美 80 年代流行音乐经典回顾",
    "'Rickroll' 网络迷因文化起源与传播",
    ...
  ]
}
```

我承认在屏幕前笑了 1 分钟——LLM 居然识别出了 "Rickroll" 这个网络梗，自动写进了 key_points 里。

这就是 AI Agent 跟传统爬虫的本质区别：

- 传统爬虫返回的是**字段**
- AI Agent 返回的是**理解**

## 七、给同样想做 AI Agent 的你

如果你也是从测试 / 运维 / 后端转型，跟我一样卡在"想做但不知道从哪开始"，我能给的建议：

**1. 先想清楚"4 层"，再写一行代码**

别上来就 LangChain / LangGraph / AutoGen。先用最朴素的 Python 类把 Monitor / Collector / Processor / DB 这 4 层手写一遍。等你**亲手感受过状态机怎么连接它们**，再去看那些框架，你会发现它们解决的就是你刚才用 if-else 解决的问题。

**2. 选对 LLM，别在 OpenAI 上烧钱**

国内开发者首选**火山方舟 Coding Plan**：100 元/月、无限调用、kimi-k2.6 / glm-5.x 任选。
我做完整个 v1.0 + v1.1，**LLM 总成本 0 元**（包月内）。

**3. SQLite 比你想的好用**

不要被"数据库"三个字吓到。SQLite 是单文件、零配置、Python 内置。
4 张表 + 5 种状态，足够支撑一个真实的 AI Agent。

**4. 状态机思维比代码本身重要**

我项目能在第二天加上"自动重试失败任务"的能力，**完全因为 v1.0 时已经把状态画对了**：

- raw_contents 是单独表（LLM 失败不用重新爬）
- task_queue.status 是游标（标记进度，不存数据）
- 失败任何一步都能从断点继续

这不是巧合，这是设计。

**5. 写 RETROSPECTIVE，比写代码更重要**

我每完成一个版本都强迫自己写一份复盘报告。不是给别人看的，是给**未来的自己**看的——半年后我重新打开这个项目，看到 RETROSPECTIVE 里写的 "B 站 API 加了 WBI 签名"，能立刻找回上下文。

代码会过期，文档不会。
