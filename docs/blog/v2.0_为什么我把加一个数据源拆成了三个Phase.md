# 为什么我把"加一个数据源"拆成了三个 Phase

> v2.0 重构复盘：一个 AI 采集项目从单源到插件架构的三段式工程笔记

[v1.0 起步篇](https://github.com/nakajimamiyuki/ai_collector_project/tree/v1.0) · [v1.1 加固篇](https://github.com/nakajimamiyuki/ai_collector_project/tree/v1.1) · [v2.0 完整代码](https://github.com/nakajimamiyuki/ai_collector_project/tree/v2.0)

---

## 楔子：一个看起来不该是 3 个 Phase 的需求

v1.1 上线后第三天，我打开 v2.0 待办清单，最上面一条只有几个字：

> 多源拓展（B 站之外接一个新源）

按字面意思，这事不复杂。`BiliMonitor` 已经能抓 B 站 UP 主了，再写一个 `ArxivMonitor` 抓 arXiv 论文，10 行代码、半小时搞定。

但我没这么干。

我把这件事拆成了**三个 Phase**，前后做了三天，每个 Phase 一个独立 commit，每个都能单独回滚、单独讲清楚。

为什么？

**因为"加一个源"和"让项目能持续加源"是两件事。**

前者是写代码，后者是改架构。我想要后者。

这篇文章把这三个 Phase 一段段拆给你看——不是为了炫技，是想把"为什么这样拆值得"讲明白。如果你做过项目、心里也有过"看起来 10 行代码就能改完，但好像应该认真重构一下"的纠结，希望这篇能给你一个清晰的答案。

## 一、v1.1 留下的真实问题

先把 v1.1 的代码现状摆出来，不藏。下面是 v1.1 时期 `monitor.py` 的开头：

```python
# v1.1 src/monitor.py
class BiliMonitor:
    API_URL = "https://api.bilibili.com/x/space/arc/search"
    # ... 写死的 B 站 headers、cookie、错误码、Playwright fallback ...

    async def sync_targets_async(self, target_uids):
        for uid in target_uids:
            urls = await self.fetch_bilibili_urls_async(uid)
            ...
```

类名叫 `BiliMonitor`，参数叫 `target_uids`（B 站 UID），方法叫 `fetch_bilibili_urls_async`——名字、参数、内部实现，**从上到下都写死了 B 站**。

processor 也一样：

```python
# v1.1 src/processor.py
def _build_prompt(self, markdown_text):
    return f"""你是一个专业的数据分析师。请从以下 B 站视频页面的 Markdown 文本中精准提取关键信息...
要求字段：
- title: 视频标题
- up_name: UP 主名称
- play_count: 播放量
- danmaku_count: 弹幕数量
- tags: 视频标签
...
"""
```

prompt 里 `up_name`、`play_count`、`danmaku_count`、"B 站视频"——全是 B 站语义，硬编码进字符串模板。

collector 更不用说，整个文件都是 `BiliCollector`，专门用 Playwright 抓 B 站 SPA 的 DOM。

**问题不在"代码不工作"——v1.1 跑得好好的。问题在"代码的形状"**：

| 我想做 | v1.1 的代码反应 |
|---|---|
| 接 arXiv | 复制 `BiliMonitor` → 改个名 → 再复制 `BiliCollector` → 改一遍 → prompt 字符串再来一份... |
| 接知乎 | 同上，再复制一遍 |
| 接 5 个源 | 5 份复制粘贴的 Monitor，5 份硬编码的 prompt |

这就是经典的**写死了的代码** —— 能跑，但每加一个新东西都要付一次"复制粘贴税"。

v2.0 要做的不是"加一个源"，是**把这个税废掉**。

## 二、Phase 1 · 抽象：零破坏的纪律

> commit [`b9e19aa`](https://github.com/nakajimamiyuki/ai_collector_project/commit/b9e19aa)（Phase 1 部分）

第一步要做的事，听起来反直觉：**不引入任何新功能，只把代码"形状"改对**。

具体改动只有两件：

### 2.1 抽出 BaseSource 基类

新建 `src/sources/base.py`：

```python
# src/sources/base.py
from abc import ABC, abstractmethod
from typing import List

class BaseSource(ABC):
    """所有信息源的抽象基类。"""

    source_type: str = "base"   # 子类必须覆盖（如 "bilibili"、"arxiv"）

    @abstractmethod
    async def fetch_new_urls(self) -> List[str]:
        """返回该源当前可见的内容 URL。失败时返回 []，不抛业务异常。"""
        raise NotImplementedError
```

只有一个抽象方法 `fetch_new_urls()`。这是所有信息源唯一要做的事：**发现 URL**，去重和入库交给上层。

然后把 v1.1 的 `BiliMonitor` 整体搬到 `src/sources/bilibili.py`，改名 `BilibiliSource`，**抓取逻辑一行都不改**（requests 主路径 + Playwright fallback + cookie 注入 + 业务错误码识别），只是包成 `BaseSource` 接口：

```python
# src/sources/bilibili.py
class BilibiliSource(BaseSource):
    source_type = "bilibili"

    def __init__(self, uids: List[str]):
        self.uids = uids
        # ... 把 v1.1 BiliMonitor 的 __init__ 内容原样搬过来 ...

    async def fetch_new_urls(self) -> List[str]:
        all_urls = []
        for uid in self.uids:
            urls = await self._fetch_urls_for_uid(uid)
            all_urls.extend(urls)
        return _dedup_keep_order(all_urls)

    # _fetch_urls_for_uid、_fetch_via_requests、_fetch_via_playwright_async
    # ... 全部从 v1.1 BiliMonitor 原样搬过来 ...
```

### 2.2 Monitor 退化为调度器

`src/monitor.py` 几乎被推倒重写——但**对外接口签名一行不变**：

```python
# v2.0 src/monitor.py
class Monitor:
    """信息源调度器：持有一组 BaseSource，逐个发现新 URL 并入库。"""

    def __init__(self, sources: Optional[List[BaseSource]] = None):
        self.db = DBManager()
        self.sources: List[BaseSource] = list(sources) if sources else []

    async def run_all_async(self) -> int:
        total = 0
        for source in self.sources:
            try:
                urls = await source.fetch_new_urls()
            except Exception as e:
                logger.error(f"source {source!r} raised: {e}")
                continue
            if urls:
                total += self.db.add_new_urls(urls)
        return total

    # ---- 向后兼容入口：旧调用 / cron 不用改 ----
    async def sync_targets_async(self, target_uids) -> int:
        if not self.sources and target_uids:
            self.add_source(BilibiliSource(uids=target_uids))
        return await self.run_all_async()


# 旧代码 `from src.monitor import BiliMonitor` 仍可用
class BiliMonitor(Monitor):
    """[DEPRECATED] v1.1 类名，保留为薄别名。"""
    pass
```

注意最后那个 `BiliMonitor(Monitor)` 别名——这是关键。`main.py`、cron 脚本、任何引用了旧名字的地方，**一行都不用改**就能继续工作。

### 2.3 这一步的纪律：零破坏 + 零新功能

Phase 1 整体看下来：
- 加了一个抽象基类
- 把 v1.1 的 B 站逻辑搬到子类，**没改一行抓取代码**
- 新 Monitor 调度多个 Source，**保留 v1.1 全部公共接口**
- 数据库 schema **完全不动**

跑一遍 main.py，发现新 URL、采集、LLM 清洗、入库——和 v1.1 完全一致的行为。

这就是**零破坏重构**：**功能 0 变化，但形状变了**。一行新功能没加，但已经为后面加任何新源做好了准备。

> 💡 **工程教训：重构和加功能必须分开提交。**
> 把"我把代码形状改对了"和"我加了 arXiv 源"放在一个 commit 里，review 时无法区分哪些改动是"重构"（应该零行为变化）、哪些是"新功能"（应该有可验证的新行为）。混在一起，万一新功能有 bug，回滚就等于丢掉了整个重构成果。

## 三、Phase 2 · 接源：用真实新源验证抽象

> commit [`b9e19aa`](https://github.com/nakajimamiyuki/ai_collector_project/commit/b9e19aa)（Phase 2 部分）

到了 Phase 2，我才开始真正"加 arXiv"。

但这一步的**真实目的不是"接到了 arXiv"，是"用一个真实的新源把 Phase 1 的抽象验证一遍"**——如果 Phase 1 抽象错了，接源时会感到别扭，要回头改基类。

事实是没返工。但中间冒出来几个有意思的"异质源"挑战。

### 3.1 ArxivSource 的实现

```python
    source_type = "arxiv"

```

`fetch_new_urls()` 接口完美贴合——不用改 BaseSource、不用改 Monitor，**直接 `add_source(ArxivSource(...))` 就能接进流水线**。

`main.py` 的改动只有这两行：

```python
self.monitor = Monitor(sources=[
    BilibiliSource(uids=self.TARGET_UIDS),
    ArxivSource(categories=["cs.AI", "cs.CL", "cs.LG"], max_results=5),
])
```

**这一刻 Phase 1 的抽象就被验证了**——加一个完全异质的源，主流程 `run()` 一行没改。

### 3.2 但异质源带来的真实挑战

接完发现：**接源很顺，但下游有连锁反应**。

**挑战 1：collector 怎么抓 arxiv？**

B 站是 SPA，必须 Playwright + stealth 才能拿到 DOM。arxiv 是服务端渲染的静态 HTML，requests + BeautifulSoup 几百毫秒就拿到了——再开浏览器是浪费。

所以 `collector.collect_content(url)` 必须按源选策略：

```python
async def collect_content(self, url, source_type=None):
    if source_type is None:
        source_type = "arxiv" if "arxiv.org" in url else "bilibili"

    if source_type == "arxiv":
        return self._collect_arxiv(url)     # requests，~0.5s
    return await self._collect_bilibili(url)  # Playwright，~10s
```

**挑战 2：processor 怎么用 LLM 提取 arxiv 字段？**

B 站 prompt 里有 `up_name`、`play_count`、`danmaku_count`——arxiv 论文没这些。论文有作者、有学科分类、有摘要——但这些 B 站 prompt 里没提。

直接把 arxiv 内容塞进 B 站 prompt 会怎样？我试了——LLM 会硬填 `play_count: null, danmaku_count: null`，summary 还可能输出英文（因为 arxiv 摘要是英文）。**字段语义错位**，不是 production 级。

解法：按 source_type 选 prompt：

```python
def _build_prompt(self, markdown_text, source_type="bilibili"):
    if source_type == "arxiv":
        return self._build_arxiv_prompt(markdown_text)
    return self._build_bilibili_prompt(markdown_text)
```

arxiv prompt 里专门写明"summary 和 key_points 必须用中文输出，即使原文是英文"。

### 3.3 验证

跑一次双源 pipeline：

```
Registered sources: ['bilibili', 'arxiv']
[Monitor] bilibili: 40 urls -> 0 new
[Monitor] arxiv: 13 urls -> 13 new tasks added
[Collector] Collecting (arxiv) from: https://arxiv.org/abs/2606.20560
[Collector] OK (arxiv) | len=2191 title=Y abstract=Y   ← 0.5 秒
```

13 篇论文入库，arxiv 采集 0.5 秒（没开浏览器），LLM 输出中文 summary——完全符合预期。

> 💡 **工程教训："再做一个"是检验"抽象是否正确"的唯一可靠方法。**
> Phase 1 时我以为抽象对了，但只有 Phase 2 接了一个真实异质的新源、并把 collector/processor 的连锁反应都跑通，才能说"这个抽象站得住"。一次不算抽象，两次才算。这也是为什么 Phase 1 不能跳过、直接和 Phase 2 一起做——你得先有一个**只重构、不加功能**的干净基线，才能在 Phase 2 用新源去打靶。

## 四、Phase 3 · 字段化：让正确的事情更显式

> commit [`d16ae54`](https://github.com/nakajimamiyuki/ai_collector_project/commit/d16ae54)

走到 Phase 3，我已经有了一个能跑的双源系统。但代码里有一处**临时方案**让我不舒服：

```python
# Phase 2 时 source_type 是这样推断的
if source_type is None:
    source_type = "arxiv" if "arxiv.org" in url else "bilibili"
```

`source_type` 没在 DB 里——每次要用就靠 URL 字符串匹配现猜。这能跑，但有三个问题：

1. **隐式而非显式**：URL 推断散落在 collector、processor 多处，规则不集中
2. **加新源就要改多处**：接知乎要改 collector 的 `if`、processor 的 `if`、所有未来用到的地方
3. **下游分析受限**：想按来源统计入库数、想让飞书报告按"📺视频/📄论文"分组——SQL 里没字段，得 LIKE 模糊匹配

Phase 3 要做的事：**让 `source_type` 进 DB，成为一等公民**。

### 4.1 schema 变更 + 幂等迁移

task_queue 和 final_results 加 `source_type` 字段：

```sql
ALTER TABLE task_queue   ADD COLUMN source_type TEXT DEFAULT 'bilibili';
ALTER TABLE final_results ADD COLUMN source_type TEXT DEFAULT 'bilibili';
```

但这是动**真实数据库** —— 我本地库里已经有 v1.x 时期的 46 条 B 站任务 + Phase 2 入库的 13 条 arxiv 任务。怎么办？

写一个**幂等迁移**方法，每次启动安全调用：

```python
def _migrate_v2_0(self):
    """
    v2.0 schema migration：加 source_type，回填老数据。
    幂等：检查字段是否已存在，存在就跳过。
    """
    with self._get_connection() as conn:
        cursor = conn.cursor()
        cursor.execute("PRAGMA table_info(task_queue)")
        cols = {row["name"] for row in cursor.fetchall()}

        if "source_type" not in cols:
            cursor.execute(
                "ALTER TABLE task_queue ADD COLUMN source_type TEXT DEFAULT 'bilibili'"
            )
            # 回填：老库里已有的 arxiv 任务
            cursor.execute(
                "UPDATE task_queue SET source_type = 'arxiv' "
                "WHERE url LIKE '%arxiv.org%'"
            )
            logger.info(f"backfilled {cursor.rowcount} arxiv rows")
        # final_results 同理...
```

幂等是关键——程序每次启动都会调它，必须保证"已经迁移过的库再跑一次不会出错、不会重复改"。

### 4.2 动 schema 前先备份真实库

这是个**操作纪律**，写进流程：

```bash
cp data/collector.db data/collector.db.bak_phase3
# 跑迁移 + 验证
# 验证通过才删备份
rm data/collector.db.bak_phase3
```

不要相信"我的迁移代码一定对"——破坏性操作前的备份是 30 秒的事，省下的可能是几小时的数据恢复。

### 4.3 流程分流从"猜 URL"升级为"读字段"

迁移完成后，collector 和 processor 改成**优先用显式 source_type**，URL 推断降为兜底：

```python
def clean_data(self, markdown_text, url=None, source_type=None):
    # 优先用显式 source_type；否则按 URL 兜底推断
    if source_type is None:
        source_type = "bilibili"
        if url and "arxiv.org" in url:
            source_type = "arxiv"
    prompt = self._build_prompt(markdown_text, source_type=source_type)
    ...
```

main.py 的 stage2/stage3 从 DB 读出每条任务的 source_type，显式传下去：

```python
cursor.execute(
    "SELECT url, source_type FROM task_queue WHERE status='PENDING' LIMIT ?",
    (max_count,),
)
for url, source_type in pending:
    content = await self.collector.collect_content(url, source_type=source_type)
```

### 4.4 验证

跑一遍真实 pipeline，task_queue 现在能这样查：

```
arxiv     | PENDING    | 7
arxiv     | COLLECTED  | 6
bilibili  | COMPLETED  | 44
bilibili  | FAILED     | 1
```

按来源 + 状态分组——SQL 一句话能拿到的东西，Phase 2 时要靠 URL LIKE 才能模糊出来。

> 💡 **工程教训：显式优于隐式，但要在合适的时机引入。**
> 如果 Phase 1 时就给 task_queue 加 `source_type` 字段，会怎样？那时只有 B 站一个源，字段永远是 `bilibili`——没有区分价值，纯粹是"为将来准备的代码"。等真的接了第二个源、跑了一阵、确认需要按源区分了，再让它进 DB——这一次 schema 变更才花得值。**架构能力不是"提前埋下所有未来的钩子"，是"在合适的时机让正确的事情更显式"。**

## 五、收尾 · 给项目加上单元测试

> commit [`272addf`](https://github.com/nakajimamiyuki/ai_collector_project/commit/272addf)

v2.0 的最后一步，不是新功能，也不是重构——是把项目的"测试质量"补到工程化的标准线上。

### 5.1 之前的"测试"到底测了什么？

v1.x 时期项目根目录里有 8 个 `test_*.py` 文件。打开 `test_arxiv_e2e.py` 看看：

```python
# 旧风格的"测试"
async def main():
    src = ArxivSource(categories=["cs.AI"], max_results=2)
    urls = await src.fetch_new_urls()      # 真实访问 arxiv.org
    assert urls
    md = await collector.collect_content(urls[0])  # 真实抓页面
    assert md
    result = proc.clean_data(md, url=urls[0])      # 真实调 LLM
    print(result)
```

这些脚本能跑，能给我"看起来都工作了"的信心。但严格说**它们不算单元测试**：

- 离线跑不了——arxiv API 不通就 fail
- LLM API 抽风也 fail——但 fail 的原因和我代码无关
- 没法进 CI——CI 环境哪有 BILI_COOKIE，哪有 LLM API_KEY
- 跑一次要几十秒——慢到没人会频繁跑

它们测的是"**今天网络好不好、LLM 心情好不好**"，不是"**我的代码对不对**"。

### 5.2 真正的单元测试：mock 掉外部世界

`tests/` 目录下重新写了 4 个测试文件，36 个 case，0.3 秒跑完，全部离线：

```
tests/
├── conftest.py                    # temp_db 等共享 fixtures
├── test_db_manager.py             # 14 个：建表/迁移/状态机/重试
├── test_sources_arxiv.py          #  6 个：mock requests，验证 XML 解析
├── test_processor.py              # 10 个：JSON 解析健壮性 + prompt 分流
└── test_collector_dispatch.py     #  6 个：分派逻辑，mock Playwright
```

关键技巧是**用 mock 把外部依赖替换成可控的**：

```python
# tests/test_sources_arxiv.py
@pytest.mark.asyncio
async def test_fetch_handles_network_exception():
    """requests 抛异常时应返回空列表，不让流水线崩。"""
    src = ArxivSource(categories=["cs.AI"])
    with patch("src.sources.arxiv.requests.get",
               side_effect=requests.RequestException("network down")):
        urls = await src.fetch_new_urls()
    assert urls == []
```

这个测试**没碰任何真实网络**——`requests.get` 被 patch 成"一定抛异常"。我在测的是"我的代码遇到网络异常时的容错路径"，不是"网络通不通"。

DB 层更彻底——每个测试一个独立的临时 SQLite 文件：

```python
# tests/conftest.py
@pytest.fixture
def temp_db(tmp_path):
    db_path = str(tmp_path / "test_collector.db")
    return DBManager(db_path=db_path)
```

测完 pytest 自动清理 `tmp_path`，连真实库 `data/collector.db` 都不碰一下。

### 5.3 现在敢动代码了

36 个测试 0.3 秒跑完——意味着我每改一行代码都能"顺手跑一遍测试"。

```
$ pytest tests/ -q
............................................
36 passed in 0.28s
```

这就是单元测试的真实价值——**不是为了"测试覆盖率好看"，是让你敢改代码、敢重构、敢加功能**。

> 💡 **工程教训：单元测试的本质是把外部世界 mock 成可控的。**
> 测试的对象永远是"你自己的逻辑"，不是"网络/LLM/浏览器是否可用"。需要真实环境跑通的那些（v1.x 那 8 个脚本）也有价值，叫"冒烟测试 / 集成测试"，应该和单元测试并存。但每次写代码改代码时频繁跑的、能进 CI 的、应该是**纯本地、毫秒级、和外部世界完全无关**的单元测试。

## 六、三段式 vs 一锅烩：为什么这样拆值得

文章开头我说"加一个源 10 行代码就能搞定"——这是真的。如果我那么做，会得到这样一个 commit：

```
feat: 接入 arXiv 源
  src/monitor.py      | 50 ++++++++-
  src/collector.py    | 80 ++++++++++++-
  src/processor.py    | 60 ++++++++++-
  src/db_manager.py   | 30 ++++++-
  main.py             | 20 ++++-
  data/collector.db   | <schema 变化>
```

能跑，但会带来这些隐性损失：

| 维度 | 一锅烩 | 三段式 |
|---|---|---|
| Code review | 抽象 / 新功能 / schema 变更 / 老数据回填 全混在一起，review 时无法分头判断 | 三个独立 commit，每段聚焦一个意图 |
| 回滚成本 | 中间任何环节出问题 → 回滚等于丢掉全部成果 | 任何一段可单独回滚，前面的不受影响 |
| 验证粒度 | 一次跑通全部 → 任何中间假设错了都难定位 | 每段独立验证（Phase 1 跑 v1.1 行为、Phase 2 跑双源、Phase 3 跑分流） |
| 讲述成本 | "我重构了项目并接了 arxiv 还改了数据库..." | "Phase 1 抽象 → Phase 2 验证 → Phase 3 显式化" |
| 风险 | 半完成状态留在主分支 | 每段都是稳定可发布的中间态 |

第 5 行是面试时最值钱的——把一坨改动讲清楚很难，把一个三段式叙事讲清楚很容易。

更重要的是：

**"重构 → 验证 → 显式化"这个三段式，不是 v2.0 一次性的方法，是处理"我想让架构能持续扩展"这类需求的通用模板。**

下次你遇到"加第 N 个东西，但代码形状不支持"的需求，可以试试这个三段：

1. **重构**：把"形状"改对，零行为变化，零新功能
2. **验证**：用第一个真实新需求验证抽象是否正确（不对就退回 1）
3. **显式化**：等抽象稳了、用过几次了、知道哪里需要显式化了，再让它进 DB / 进类型 / 进配置

每一段一个 commit，每一段独立可回滚，每一段独立可讲述。

## 七、结语 · 给一年前的自己写一封信

如果让我用一句话总结这个项目走到 v2.0 的三步：

> **v1.0 让它跑、v1.1 让它稳、v2.0 让它能持续生长。**

v1.0 时我写第一版 `BiliMonitor`，写完就觉得"行了能跑了"。如果一年前的我看到这次 v2.0 的拆分，大概会想"至于吗？不就加个源吗？"。

但现在我知道，**项目的真正价值不在"功能多"，在"加下一个功能的成本是否还能控制"**。

v1.x 的 `BiliMonitor` 能跑，但每加一个新源都是一次复制粘贴税。v2.0 把这个税废掉了——加 arxiv 用了一天，下次加知乎可能就半天，再加掘金可能就一上午。**架构的复利是这样来的。**

至于 v3.0 会做什么——可能是把"飞书每日报告按来源分组"做完整，可能是接微信公众号源，可能是把整套搬到 LangGraph 上做工作流编排。不急。
