#### 2.2.1 `query_rag(question, top_k)` —— 让 MCP server 能听人话

字面匹配的弱点很明显：

```text
查："会用 LangGraph 做反思决策的 Agent 岗位"
search_jobs → 返回 0 条（JD 里根本没人写"反思决策"四个字）
```

但有一条 JD 写了：

> "基于 LangGraph、LangChain、AutoGen 等框架开发 Agent 应用，**运用 CoT（Chain-of-Thought）** 实现复杂业务流程的自动化"

——"运用 CoT" 就是反思决策的语义近义词。这种召回，**只有向量检索能做到**。

实现上，`query_rag` 直接复用 v3.0 已有的 `src/agent/tools.py::vector_search_jobs`：

```python
@mcp.tool()
def query_rag(question: str, top_k: int = 5) -> str:
    """用自然语言对 Boss 直聘岗位做向量检索（bge-m3 + Milvus）。"""
    from src.agent.tools import vector_search_jobs
    try:
        records = vector_search_jobs(question, top_k=top_k)
    except Exception as e:
        # ollama 没起 / vector.db 不存在 → 返回带 hint 的错误 JSON
        # 而不是抛异常，让 MCP 客户端能直接拿到可读信息
        return json.dumps({"error": ..., "hint": "ollama serve && ...", "hits": []})
    ...
```

这里有个细节：**基础设施故障要翻译成 JSON 错误，不能让异常穿透到客户端**。MCP 客户端拿到 `isError=True` 的 content 时，用户体验是 "工具坏了不知道为啥"。我让它返回 `{"error": "...", "hint": "..."}`，客户端直接能告诉用户「去启动 ollama」。

跑出来的真实例子：

```
query_rag("会用 LangGraph 做反思决策的 Agent 岗位", top_k=3)

[0.7974] Python(LangChain/LightRAG/AI Agent) | 深圳中迈 | 20-25K | 杭州
[0.7872] 高级 AI Agent 开发工程师             | 堃垚集团 | 13-25K | 郑州
[0.7851] agent开发-乾健科技                   | 武汉添才 | 20-35K | 济南
```

第三条 JD 写了"基于 LangGraph、LangChain、AutoGen 等框架开发 Agent 应用，运用 CoT"——**字面搜不到"反思决策"，但语义检索召回了**。这条数据本身就是面试现场最好的 demo。

## 三、两个把我卡住 20 分钟的子进程坑

中间踩了两个非常有代表性的坑，值得单独写。

### 3.1 坑 1：`ModuleNotFoundError: No module named 'src'`

写完 query_rag 和 get_skill_gap，先跑单元测试——全绿。然后跑真实的 stdio MCP client smoke test：

```python
async with stdio_client(server) as (read, write):
    async with ClientSession(read, write) as session:
        await session.initialize()
        # search_jobs 跑通
        r = await session.call_tool('search_jobs', {...})
        print(r.content[0].text)  # ✅ 5 matches
        # get_skill_gap 也跑
        r = await session.call_tool('get_skill_gap', {...})
        print(r.content[0].text)  # ❌ Error executing tool get_skill_gap: No module named 'src'
```

**单元测试全绿、search_jobs 跑通、新工具调用就报错。**

把异常拿出来看，`get_skill_gap` 里有这一行：

```python
from src.agent.tools import compute_skill_gap, load_profile
```

而 `search_jobs` 没有这种延迟 import，所以它能跑。

为什么 venv 里跑直接 import 是好的，子进程里就不行？

> **`python src/mcp_server/ai_collector_mcp.py` 当脚本运行时，Python 默认只把脚本所在目录（`src/mcp_server/`）加进 `sys.path`。**
> **顶层的 `src` 包根本不在 import path 里，所以 `from src.agent.tools import ...` 找不到。**

单元测试时是 pytest 启动的，pytest 默认会把项目根加进 sys.path（`conftest.py` 或 `pytest.ini` 的 `rootdir` 机制），所以测试看不到这个坑。**只有真实子进程模式才暴露**。

修复：

```python
PROJECT_ROOT = Path(__file__).resolve().parents[2]
if str(PROJECT_ROOT) not in sys.path:
    sys.path.insert(0, str(PROJECT_ROOT))
```

教训记下来：**写 MCP server 这种"会被以子进程启动的模块"，永远要在文件顶部手动注入项目根**。这个坑跟早期写 cron 脚本时遇到的 PATH/PYTHONPATH 问题是同一类。

### 3.2 坑 2：macOS OpenMP 双库 abort

修完坑 1，再跑：

```
✅ search_jobs OK
✅ get_skill_gap OK（不走 milvus，纯 SQLite）
❌ query_rag → McpError: Connection closed
```

为什么连接被关掉？看 stderr 输出：

```
INFO     Loading faiss.
INFO     Successfully loaded faiss.
INFO     MilvusLite server started for ... on port 61160
OMP: Error #15: Initializing libomp.dylib, but found libomp.dylib already initialized.
OMP: Hint This means that multiple copies of the OpenMP runtime have been linked into the program.
```

这个错我 v2.1 时就踩过，所以 v3.0 的 `scripts/*.py` 顶部都设了 `KMP_DUPLICATE_LIB_OK=TRUE`。**但今晚新写的 MCP server 入口我忘了设。**

根因：

> macOS 上 `milvus-lite` 内嵌的 `faiss` 链了一份 `libomp.dylib`，`numpy` / `torch` 也各自链一份，**同一进程加载多份会触发 OMP 运行时的安全检查，直接 abort**。
> abort 之后 stdout 已经断了，stdio MCP client 看到的是 `Connection closed`，根本想不到是 OpenMP 的问题。

修复就一行：

```python
os.environ.setdefault("KMP_DUPLICATE_LIB_OK", "TRUE")
```

但**必须在 `import faiss/milvus` 之前设**，所以放在文件最顶部 `import` 之前。

为什么写这两段？因为转型期最浪费时间的不是"不知道答案的问题"，是"答案被 5 层抽象隐藏的问题"。MCP client 报 `Connection closed`，你怎么也想不到是 OpenMP 库；`No module named 'src'`，你怎么也想不到跟 pytest rootdir 机制有关。

**写代码的工程师能力分水岭，不在算法，在能不能 5 分钟内把故障定位到这一层**。这两个坑都是我从 stderr 一行一行往回追溯的，每一步都没快捷方式。这个能力比"懂 LangGraph 框架"对实际工作更值钱。

## 四、F2 ── Bad Case 闭环：让 Agent 自己变好

### 4.1 这个想法是怎么来的

v3.0 跑通那天我跑了两个 query：

```
Run 1: "LangChain RAG 15K+ 1-3 年" → 6 条命中 ✅
Run 2: "郑州 25K+ LangGraph"      → 3 轮反思后 0 命中（故意稀有）❌
```

Run 2 看着很有"反思决策"的戏剧感（错配诊断 + 替代方案），但**这两次跑完，下次我打开项目就忘了**。没有任何持久化记录。

下次再跑同一个 query 时：

- 我不知道上次跑过没
- 我不知道上次结果是不是空
- 我不知道上次空了之后我有没有修过什么
- 我不知道这次修完之后是不是真的修好了

这就是典型的 **没有 bad case 闭环** 的 Agent 项目。市面上 90% 的 LangGraph demo 都是这个问题——能跑一次给视频录屏用，但不能形成"跑 → 复盘 → 修 → 验证"的工程闭环。

而这一项，是我**最该有的能力**——我做了 2 年第三方测评，军工/航天级 CNAS+CMA 测评的核心就是这套。**只是之前对象是硬件，现在换成 LLM**。

### 4.2 设计：物理隔离 + 零结果自动 bad

第一个决策：**bad case 库要不要塞进 `collector.db`？**

不。

`collector.db` 是 v1/v2 pipeline 的状态机（task_queue / raw_contents / final_results），schema 已经稳定，加新表会污染语义边界。

```
collector.db        → v1/v2 pipeline 状态机（采集 / 清洗 / 入库）
                      表：task_queue / raw_contents / final_results

agent_runs.db       → v3+ Agent 维度数据（独立物理文件）
                      表：agent_runs
```

两个库都是 SQLite，但解耦：未来想把 agent_runs 单独导出 / 迁移 / 对外暴露，零成本。

第二个决策：**零结果是不是自动 bad？**

是。

```python
def record_run(self, *, query, result_count, ...):
    ...
    if result_count == 0 and status == "unreviewed":
        status = "bad"
        auto_bad_reason = "zero_result"
```

为什么这么设计？因为**"什么算 bad case"这个问题，大部分情况下不需要人来判断**。Agent 跑出来 0 条结果，95% 的概率就是 bad case。**让机器先标，人只复核**——这跟我以前做测评写自动化测试用例的习惯一模一样：先让测试用例自己判生死，人只看 fail 的。

第三个决策：**replay 命令必须有**。

```bash
python scripts/agent_runs.py replay --status bad --limit 5
```

这是整个闭环的关键步骤——**你以为修好的 bad case，必须用同一份 query 重跑一遍验证**。不重跑就不算修好。这一条是我在测评工作里被反复教育的：「regress 不是说说的，是要重新跑测试用例的」。

5 行内塞满所有关键词，HR 扫一眼能拿到所有信号。

### 5.2 F4 ── 架构图 SVG

旧 README 里有一张 ASCII 架构图，git 仓库里能看，**但简历 PDF / 微信传送时会丢字符**。

要一张真图。

第一直觉是用 mermaid-cli（`mmdc`）。一查——没装。装它要拉 Chromium 内核，国内拉镜像可能要几分钟，可能要重试。

**避坑思路**：直接手写 SVG。GitHub README 原生支持 `<img src="docs/assets/architecture.svg">`，矢量、可缩放、文件只有 14KB。简历用时 macOS 自带的 `qlmanage -t -s 1920 -o /tmp` 一行命令导 PNG，零依赖、零拉镜像。

写完了用浏览器打开自己看，让视觉模型检查"有没有线穿过不相关的方块"。第一版有两条线穿透了关键节点，改了路径重检查——干净。

**关键经验**：写文档/图表，能让自己 5 分钟内可重做的方案，永远优先于"装套工具一劳永逸"的方案。后者一旦遇到环境问题（macOS 上的 mmdc / 国内镜像 / 版本冲突），就是 30 分钟起。

最后 README 顶部就长这样：

```html
<p align="center">
  <img src="docs/assets/architecture.svg" alt="Architecture overview" width="100%">
</p>
```

GitHub 直接渲染矢量图，简历用 `qlmanage` 导 PNG。

## 六、收尾反思：从"能跑"到"投得出去"，几乎一样难

合上电脑前再看一眼 git log：

```
v3.0 (2026-06-24) 一天 5 阶段写完 LangGraph Agent
v3.1 (2026-06-26) 一天加 3 个 MCP Tool + Bad Case 闭环 + 简历级打磨
```

但我心里清楚，**v3.0 → v3.1 这一步的工作量，跟 v0 → v3.0 几乎一样大**。

v3.0 之前我做的是「让项目长出能力」：采集层、RAG 层、Agent 层、反思循环。每一步都看得见、摸得着。

v3.1 这一天我做的是「让项目能被看见、被复用、被传播」：

- MCP server 让别的 AI 客户端能调用我
- Bad case 闭环让我能持续把 Agent 调得更准
- 简历级 README + 架构图 + JD 对照表让 HR 30 秒看懂

**这些工作没有一行代码"能跑出新功能"，但它们决定了项目能不能用进面试、用进简历、用进真实求职**。

这个对照让我想起一句话：

> 95% 的开源项目死在"能跑但讲不清楚"。

我现在 26 岁，从测评转 AI 应用开发，时间窗口只有今年下半年。**项目骨架在 v3.0 已经建完，v3.1 这一晚做的就是把它打磨到"投得出去"的程度**。

明天早上我会做两件事：

1. 用 `asciinema` 录一段 v3.0 的 demo，30-60 秒，上传到 asciinema.org，README 嵌入
2. 上 GitHub 看看 README 在线渲染效果，确认 SVG 架构图正常显示

然后这个项目就**冻结**了。

冻结不是放弃。冻结是为了进下一个阶段——

**把 v3.0+v3.1 当成"简历主项目"，反推接下来 3 个月学什么**。我的 Agent 已经告诉我答案了：

```
多模态 41 > LangGraph 16 > MCP 14 > LoRA 11
```

- MCP **稀缺讲**（v3.1 已用，市场会的人少）
- 多模态 **市场讲**（市场最缺）

够了。9 月份去新城市投简历，这四张牌打出去，AI 应用 / Agent 开发岗能进面试。
