## 楔子：项目叫 AI Agent，但其实只是 pipeline

v2.0 收完那天，我把项目从头到尾看了一遍，老实承认了一个事情——

我这个项目的名字叫 **AI Collector**，定位写的是 **"AI Agent"**。但实际架构是：

```
项目名字叫    AI Collector       这是"采集"
定位定为     AI Agent           这是"智能体"

实际架构是   Pipeline
            （监控 → 采集 → LLM 清洗 → 入库 → 推送）
                            ↑↑↑
                  LLM 只是"被调用的函数"，没有决策权
```

简历上写"AI Agent"，面试官真问一句"Agent 体现在哪？"我大概只能答"用了 LLM 做 JSON 抽取"——这撑不起 Agent 这个词。

要升级成真正的 Agent，最关键的缺口是**语义检索**。

为什么？因为 Agent 需要"自己查数据、自己决定下一步"——而我现在数据库里那 47 条结构化内容**只能用 SQL 精确查**。LLM 想问"最近关于 RAG 的内容有哪些"——它不会知道应该 `WHERE tags LIKE '%RAG%'`，它需要的是"我用自然语言问，系统给我语义相关的结果"。

这就是 RAG（Retrieval-Augmented Generation）的核心：**给数据建一个向量索引，让自然语言查询变成可能。**

今晚就把这一层补上。

## 一、选型：为什么不用 Chroma，为什么选 Milvus Lite + bge-m3

### 1.1 向量库的选择

打开任何 RAG 教程，第一段代码大概率都是：

```python
import chromadb
client = chromadb.Client()
```

Chroma 在 2024-2025 火过，因为它**装起来最简单**（pip 一行）。但 2026 年再看，我老实评估了一下：

> **Chroma 在简历上的价值在下降**——所有跟过 RAG 教程的人都写过 Chroma，面试官早看烦了。生产环境里 Chroma 性能也撑不起，企业基本不用它。

在中国市场环境下排了一下：

| 选项 | 简历加分 | 中国友好 | 我选择 |
|---|---|---|---|
| **Milvus Lite** | 🟢🟢🟢 | ✅✅ Zilliz 在中国有团队、文档全 | **✅ 选这个** |
| Qdrant | 🟢🟢🟢 | ✅ 但偏外企招聘 | 备选 |
| pgvector | 🟢🟢 中高 | ✅ | 工程量大，不适合"一晚上做完" |
| Chroma | 🟡 中（下降） | ✅ | ❌ 同质化严重 |
| Pinecone | 🟢 中 | ❌ 翻墙 + 外币卡 | ❌ pass |

**选 Milvus Lite 的理由**：
1. Zilliz 出品，**中国背景项目，中文文档极全**，国内大厂面试加分明显
2. `pip install pymilvus` + `MilvusClient(uri="x.db")` 就能跑，**和 Chroma 一样简单**
3. 完整版 Milvus 是分布式正经货——简历能写"用了 Milvus Lite，生产可平滑迁到 Milvus Cluster"
4. 阿里、腾讯、字节、蚂蚁内部都在用

### 1.2 Embedding 模型的选择

我本地 ollama 已经装了 `bge-m3:latest`，那就用它——零下载、零网络、零依赖。

bge-m3 的优势：
- 北京智源（BAAI）出品，**中文 embedding SOTA 之一**
- 1024 维，比 bge-small 高一档质量
- **支持中英混合**——后面"RickRoll" 这个英文词能命中中文 summary 就是靠这个

调用方式：本地 ollama 的 `/api/embeddings` 接口，HTTP 调用。

> 💡 **教训**：自学者最容易在选型阶段卡住——一打开知乎和 B 站全是"15 种向量库哪个好"。真实场景下选型只有三条标准：**简历认可度 + 上手成本 + 我的环境能不能跑**。其他 12 种讨论都是噪音。

## 二、设计：用 vertical tracer bullet 切分工作

这次我用了 TDD skill 里强调的 **"vertical tracer bullet"**（垂直曳光弹）原则——**不要一口气写完所有代码再开始测，而是一个模块跑通一个，再走下一个**。

切成 5 个 Phase，每个 Phase 跑通后停下来汇报：

```
Phase 1  环境准备 + 看清数据现状      10 分钟
Phase 2  Embedding 模型验证可用       15 分钟
Phase 3  写 embedder.py + vector_store.py  30 分钟
Phase 4  写批量索引脚本 + 真实跑通    20 分钟
Phase 5  CLI 检索 demo + 提交         15 分钟
```

最后总共花了大概 1.5 小时，但**修了 4 个真实 bug**——这些 bug 没有一个在写代码时能预料到，全是真实跑起来才暴露的。

> 💡 **教训**：水平切片（先写完所有测试再写所有实现）是新手最容易踩的坑——你**在没跑过的时候，假设了一堆接口长什么样**，结果都是错的。垂直切片每次只走一个端到端的小切片，每跑通一次都在校准下一段的设计假设。

### 4.2 VectorStore

Milvus Lite 封装，~180 行。Schema 设计：

```python
collection: ai_collector_chunks

字段：
  id           INT64        auto_id 主键
  url          VARCHAR(512) 原 final_results.url（用于回查 SQLite）
  source_type  VARCHAR(32)  'arxiv' / 'bilibili'（用于过滤）
  title        VARCHAR(512) 标题（用于展示）
  text         VARCHAR(8192) 被 embed 的完整文本
  embedding    FLOAT_VECTOR(1024)  实际向量

索引：AUTOINDEX + COSINE
```

关键设计：**一条 final_results = 一个向量**，不分 chunk。

为什么不分 chunk？YAGNI 原则——47 条数据 + 单条 1-2k 字符的 summary，整段 embed 完全够用。等数据上千条 + 单条很长时再上 chunk 不迟。

接口：

```python
class VectorStore:
    def upsert(self, *, url, source_type, title, text, embedding):
        """按 url 去重写入：先删后插（Milvus Lite 没有主键 UPSERT）。"""

    def search(self, query_embedding, top_k=5, source_type=None) -> list[SearchHit]:
        """语义检索，可按 source_type 过滤。"""
```

立刻测：11 个测试 + 用**真实 Milvus Lite + tmp_path**，不 mock。Milvus Lite 是进程内库（跟 SQLite 一样轻），mock 它反而失去对真实行为的验证。

```bash
$ pytest tests/test_rag_vector_store.py
============================== 11 passed (with caveats) ===
```

——是的，测试是过了，但**接下来的 Phase 4 暴露了真问题**。

## 五、Phase 4：批量索引 47 条数据 —— 一连串真实 bug

这是今晚最长的一段。从"以为能跑"到"真能跑"中间隔着 4 个 bug。

### Bug 1：pymilvus 装了但 Milvus Lite 没装

写完测试第一次跑：

```python
pymilvus.exceptions.ConnectionConfigException:
  milvus-lite is required for local database connections.
  Please install it with: pip install pymilvus[milvus_lite]
```

什么情况？我 `pip install pymilvus` 已经成功了啊。

原因：pymilvus 3.x 把 `milvus-lite` 作为 **optional extra**，不带 `[milvus_lite]` 不会自动装。

修：

```bash
pip install milvus-lite
```

> 💡 **教训**：第三方库的 optional dependencies 是常见坑。安装文档**永远要看最新的官方 README**——不要照抄知乎或 CSDN 上 6 个月前的教程命令。

### Bug 2：COSINE 返回的是 distance 不是 similarity

测试套件里有这一条：

```python
def test_upsert_then_search_finds_it(store):
    vec = _fake_vec(0.5)
    store.upsert(url="https://x.com/1", ..., embedding=vec)
    hits = store.search(vec, top_k=5)
    assert hits[0].score > 0.99   # 同一向量 COSINE 应接近 1.0
```

跑出来：

```
AssertionError: assert 0.0 > 0.99
```

——什么？同一个向量自己搜自己，相似度是 **0.0** ？

这里我应用了刚学的 **systematic-debugging skill** 的诊断流程，形成 3 个排序假设：

1. **假设 A**：字段名错了——Milvus 不叫 `distance` 而是别的
2. **假设 B**：我的"全 0.5"假向量太退化，COSINE 算出来确实是 0
3. **假设 C**：写入后还没 flush，索引没建好

最便宜的验证：手动跑一次，看 Milvus 原始返回是什么：

```python
raw = client.search(...)
print(raw[0][0])
# {'id': 1, 'distance': 0.0, 'entity': {...}}
```

字段名是对的。再用真正不同方向的向量测：

```
完全相同向量 → distance = 0.000000   （最近）
几乎相同向量 → distance = 0.000193   （次近）
完全相反向量 → distance = 2.000000   （最远）
```

**根因**：Milvus Lite 在 COSINE metric 下返回的是**距离**（0 = 完全同向；2 = 完全反向），**不是相似度**。我注释里写的"COSINE 越大越相似"是错的。

修：在 `search()` 里做转换，下游拿到的是更直觉的 0-1 相似度：

```python
distance = float(h.get("distance", 0.0))
similarity = 1.0 - distance / 2.0     # 0=反向, 0.5=正交, 1=同向
```

> 💡 **教训**：**先建一个能让你看到原始数据的小工具，再下结论**。直接对着 stack trace 猜 3 分钟，不如花 1 分钟跑一次 `print(raw_result)`。这就是 systematic-debugging skill 里说的"**先建 tight feedback loop，再建假设**"。

### Bug 3：Milvus collection 跨进程默认是 released 状态

索引脚本跑完了，关掉。换一个进程跑 CLI 检索——挂了：

```
MilvusException:
  Collection 'ai_collector_chunks' is in state 'released';
  call load() before search/get/query
```

之前测试套件没挂的原因：测试和索引在**同一进程内**（建表 → 立刻搜），collection 还在内存里。但生产场景下，索引脚本退出后内存释放，下次 CLI 启动新进程，**collection 默认是 released 状态，必须先 `load_collection()` 才能搜**。

修：在 `_ensure_collection()` 末尾总是调一次 `load_collection()`：

```python
def _ensure_collection(self):
    if not self.client.has_collection(self.collection):
        # ... 建表
        ...
    # 关键：跨进程时 collection 默认 released，必须 load 才能 search
    self.client.load_collection(self.collection)
```

> 💡 **教训**：测试和真实场景的差别经常藏在**进程边界**里。如果你的测试都在单进程跑，遇到这种"测试过了但 production 挂"的问题，**第一时间想想是不是进程生命周期相关**。

### Bug 4：Milvus Lite 3.x 把 db_path 当目录用

`--rebuild` 重建时报错：

```
PermissionError: [Errno 1] Operation not permitted: 'data/vector.db'
```

我以为是 macOS 权限问题，结果一看：

```bash
$ ls data/vector.db/
LOCK  collections/
```

——`vector.db` 是个**目录**！我用 `os.remove` 删目录当然失败。

Milvus Lite 2.x 是单文件，3.x 改成了目录形式（含 LOCK + collections/）。我学的教程是 2.x 时代的，照搬就掉坑。

修：用 `shutil.rmtree`：

```python
if VECTOR_DB_PATH.is_dir():
    shutil.rmtree(VECTOR_DB_PATH)
else:
    os.remove(VECTOR_DB_PATH)
```

## 八、复盘：今晚学到的 5 件事

### 1. 选型不是技术问题，是 ROI 问题

我没选 Chroma 不是因为它差，是因为**"用 Chroma 写 RAG"已经是每个跟过教程的人都会写的东西**。在简历同质化严重的转型期，**选一个稍微小众但靠谱的工具**（Milvus Lite / Qdrant）就是差异化。

### 2. Vertical tracer bullet 真的有效

我严格按"做一个跑通一个"的节奏走，每个 Phase 完都停下来汇报。结果就是——遇到 bug 时我**永远精确知道是哪一段引入的**，不会陷入"500 行新代码不知道哪里出错"的恐怖。

### 3. systematic-debugging 的"先建可观察的 loop"是真心好用

Bug 2（COSINE 距离 vs 相似度）我一开始就想猜——"是不是字段名错了"、"是不是要 flush"。但我强迫自己**先花 1 分钟跑一个 `print(raw_result)`**，看到真实返回的瞬间，3 个假设里哪个对一目了然。

> 在 debug 上花在"看真实数据"的时间，永远比花在"猜测原因"的时间更值钱。

### 4. 跨进程的 bug 永远要等真实跑才发现

Bug 3（collection released）就是这种典型——测试单进程跑得很欢，换一个 CLI 进程就挂。**这种 bug 单元测试根本测不出来**，必须靠"我真把脚本端到端跑一遍"才能暴露。

> 💡 这也是为什么我特意写了 `scripts/index_final_results.py` + `scripts/search.py` 两个独立脚本——它们是真实的"跨进程 smoke test"。

### 5. 用过的工具版本号要记牢

Bug 1（pymilvus 没自动装 milvus-lite）、Bug 4（db_path 从文件变目录），都是**版本差异**踩的坑。我学的教程是 6 个月前的，pymilvus 3.x 改了行为，照搬就死。

> **永远以官方最新文档为准。**

## 十、下一步：从"RAG 检索"到"真正的 Agent"

今晚做完是个干净的中间态——**数据 + 检索都有了**，但还缺最后一层：**自主决策**。

下一个里程碑（v3.0）：

```
LangGraph Agent
   ├── 决策：用户问"总结一下最近 RAG 相关的论文和视频"
   ├── 工具：query_db(source='arxiv', query='RAG')
   ├── 工具：query_db(source='bilibili', query='RAG')
   ├── 反思：信息够吗？要不要再查更长时间段？
   └── 综合输出 + 引用每条数据源
```
